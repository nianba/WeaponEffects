# 全长矛适配技术文档

## 1. 目标

在当前三叉戟路线已经跑通、并完成动作数据重构的基础上，将原版长矛类武器逐步接入同一套长矛系统。

当前仓库里的成功路线仍然是三段左键循环：

```text
上挑 -> 绕身回扫 -> 终结刺击
```

代码仍保留 `ForwardThrust` 这个 step，但普通左键循环从 `SpearActionScheme.FirstEnabledStepIndex = 1` 开始，因此当前不会打出旧设计里的第一段“扎”。

本轮文档以当前仓库代码为准。重点是复用已经验证过的动作、碰撞、手持武器绘制、挥扫残影、枪尖锋芒、命中反馈和右键投掷路线。

## 2. 当前代码状态

### 2.1 当前只接管三叉戟

`SpearGlobalItem.ShouldUseSpearAction` 当前通过 `SpearWeaponClassifier` 判断：

```csharp
return ModContent.GetInstance<WeaponEffectsGameplayConfig>().EnableSpearRework
	&& SpearWeaponClassifier.IsSupportedSpear(item);
```

而 `SpearWeaponClassifier.IsSupportedSpear` 当前只允许：

```csharp
return item.type == ItemID.Trident;
```

所以仓库现状是：三叉戟基底已重构，19 把长矛还没有真正接入代码。

### 2.2 动作数据已集中到 SpearActionScheme

旧的 `TridentSpearComboScheme` 已删除。当前动作表在：

```text
Core/Spears/SpearActionScheme.cs
```

`SpearActionScheme` 定义：

```text
FirstEnabledStepIndex = 1
LifetimeTicks = 26
Steps = SpearActionStep[]
```

`SpearActionStep` 现在同时挂载五类数据：

| 字段 | 职责 |
| --- | --- |
| `Timing` | 有效帧、起手结束、攻击结束、挥舞音效触发进度 |
| `Gameplay` | reach、碰撞宽度、伤害倍率、时间倍率、extraUpdates |
| `Motion` | 弧形动作或突刺动作的角度、半径、终点参数 |
| `Afterimage` | 挥扫弧面的视觉参数和碰撞采样参数 |
| `TipGlow` | 枪尖锋芒的视觉参数和碰撞延伸参数 |

这个重构后的方向是正确的：动作定义内聚在一个 step 内，后续调动作不需要再在多个 `ForStep` switch 里来回改。

### 2.3 当前实际三段循环

`SpearActionScheme` 仍定义 4 个 step：

| Step | Kind | 名称 | 当前普通左键是否启用 |
| --- | --- | --- | --- |
| 0 | `ForwardThrust` | Forward Thrust | 否 |
| 1 | `RisingLift` | Rising Lift | 是 |
| 2 | `Backsweep` | Around-Body Backsweep | 是 |
| 3 | `Finisher` | Branching Finisher | 是 |

`WeaponEffectsPlayer.ConsumeNextSpearComboStep` 使用：

```text
1 -> 2 -> 3 -> 1 -> ...
```

因此全长矛适配第一版应按当前三段循环做，不要把 step 0 当成已启用动作。

### 2.4 终结分支现状

代码中仍有 `GroundedFinisher` 和 `AirborneFinisher`，`SpearMotion` 里也保留了 `AirborneFinisherTip`。

但 `SpearMotion.SelectFinisherBranch` 当前固定返回：

```csharp
return SpearComboBranch.GroundedFinisher;
```

所以当前实战里终结段总是地面终结刺击，不会进入空中下砸。

## 3. 本轮明确取舍

第一版全长矛适配先接受以下统一处理：

- 所有长矛先共用当前三叉戟三段左键循环。
- 不为每把武器单独改 `SpearActionScheme`。
- 武器长度和贴图原点差异先不逐把调参。
- 挥扫宽度、动作强度、碰撞扩张先统一。
- 右键蓄力投掷先作为通用长矛能力保留。
- 原版特殊弹幕按刀剑系统的思路手动补发，但长矛不能使用刀剑的通用 fallback。
- 不确定的特殊弹幕先不补发，避免原版矛本体和自定义矛叠加。

第一版重点是稳定扩大接入范围，不是逐把武器精修。

## 4. 当前管线

| 模块 | 当前职责 |
| --- | --- |
| `SpearGlobalItem` | 判断是否接管武器，隐藏原版使用图，禁用原版 Shoot，创建左键 channel 或右键蓄力 projectile。 |
| `SpearWeaponClassifier` | 当前接管白名单入口，目前只允许 `ItemID.Trident`。 |
| `WeaponEffectsPlayer` | 维护长矛三段循环索引和连招重置计时。 |

| `SpearActionStep` | 单段动作数据容器，聚合 timing、gameplay、motion、afterimage、tipGlow。 |
| `SpearActionTiming` | 管理有效帧、起手段、攻击段、挥舞音效触发进度。 |
| `SpearGameplayProfile` | 管理 reach、碰撞宽度、伤害倍率、时间倍率、extraUpdates。 |
| `SpearMotionProfile` | 管理弧形动作和突刺动作的轨迹参数。 |
| `SpearMotion` | 根据 step、分支、进度计算枪柄和枪尖位置。 |
| `SpearSweepAfterimageProfile` | 单段挥扫弧面和碰撞采样数据；当前已由 step 直接持有。 |
| `SpearTipGlowProfile` | 单段枪尖锋芒数据；当前已由 step 直接持有。 |
| `SpearCollisionEnvelope` | 从 step 的 afterimage/tipGlow 读取碰撞宽度、采样和枪尖延伸。 |
| `SpearChannelProjectile` | 读取鼠标方向，维持使用状态，按 step 节奏发出 strike 和 trail。 |
| `SpearStrikeProjectile` | 负责伤害、碰撞、命中反馈、手持武器贴图绘制、挥舞音效。 |
| `SpearTrailGlowProjectile` | 负责挥扫弧面、枪杆残影、枪尖锋芒。 |
| `SpearHeldVisualMetrics` | 统一手持长矛贴图的握点、枪尖和绘制缩放。 |
| `SpearThrowChargeProjectile` | 右键蓄力投掷的持枪姿态、蓄力反馈和释放。 |
| `SpearThrowProjectile` | 右键释放后的直线投掷伤害和视觉。 |

全长矛适配不应复制动作系统。应该让所有白名单长矛共用 `SpearActionScheme`，只在武器分类、视觉配色和特殊弹幕上做扩展。

## 5. 动作节奏

当前长矛左键节奏复用刀剑的速度配置与近战攻速逻辑。

基础间隔来自 `SpearChannelProjectile.NormalSpearInterval`：

```text
基础间隔 = weapon.useAnimation * NormalSlashIntervalMultiplier / meleeAttackSpeed
```

每段再乘以 step 的时间倍率：

```text
本段间隔 = 基础间隔 * step.Gameplay.GetTimeMultiplier(branch)
```

`SpearStrikeProjectile` 和 `SpearTrailGlowProjectile` 的生命周期也使用同一时间倍率：

```text
生命周期 = SpearActionScheme.LifetimeTicks
        * step.Gameplay.GetTimeMultiplier(branch)
        * (step.Gameplay.ExtraUpdates + 1)
```

动作速度会受以下因素影响：

- 武器原始 `useAnimation`
- `WeaponEffectsGameplayConfig.NormalSlashIntervalMultiplier`
- 玩家近战攻速 `player.GetAttackSpeed(DamageClass.Melee)`
- 当前 step 的 `Gameplay.TimeMultiplier`
- `AirborneTimeMultiplier`，但当前不会进入空中分支

挥舞音效现在由 `SpearStrikeProjectile.PlaySwingSoundAtRelease` 在 `step.Timing.SwingSoundProgress` 播放，不再由 channel 在生成 strike 时直接播放。

## 7. 入口接管规则

当前入口应沿用已经存在的 `SpearWeaponClassifier`，不要再另起一套入口 resolver。

批量适配时，将 `SpearWeaponClassifier.IsSupportedSpear` 从单把三叉戟扩展为 19 把白名单：

```text
SpearWeaponClassifier.IsSupportedSpear(item)
```

不要使用以下自动判断：

```text
item.useStyle == ItemUseStyleID.Rapier
item.shoot > ProjectileID.None
item.noMelee
item.channel
```

原因是这些条件会误伤其他投射武器、链式武器和非标准长柄武器。

接管后继续保持：

```text
item.noUseGraphic = true
item.noMelee = true
item.channel = true
item.shootSpeed = 40f
item.useStyle = ItemUseStyleID.Rapier
item.UseSound = SoundID.Item1
item.shoot = ProjectileID.None
```

`item.shoot = ProjectileID.None` 是当前避免原版矛本体叠加的关键。后续补发特殊弹幕时，不要从 `player.HeldItem.shoot` 读取原值。

## 8. 视觉路线

当前验证成功的挥扫路线是：

```text
Effect: WeaponEffects/Effects/Mhd
Textures[0]: SlashTex
Textures[1]: 干净配色源贴图
```

当前代码仍固定使用：

```text
sweepTexture = MeleeEffectAssets.SlashTexture
colorTexture = MeleeEffectAssets.SpearSweepColorTexture
```

批量适配后可以新增一个只负责视觉的 resolver，例如：

```text
Core/Spears/SpearVisualProfile.cs
Core/Spears/SpearVisualProfileResolver.cs
```

但它只负责视觉，不负责动作参数，不替代 `SpearWeaponClassifier`。

第一版 visual profile 只需要：

```csharp
public readonly struct SpearVisualProfile
{
    public readonly int ItemType;
    public readonly string SweepColorTexturePath;
    public readonly Color TipGlowColor;
}
```

批量适配后 `SpearTrailGlowProjectile` 应通过 `_weaponItemType` 解析：

```text
colorTexture = profile.SweepColorTexturePath
tipGlowColor = profile.TipGlowColor
```

不要把动作长度、挥扫宽度、碰撞强度、每把武器独立动作倍率放入 visual profile。当前动作参数统一由 `SpearActionScheme` 管。

不要回到以下方案：

| 方案 | 问题 |
| --- | --- |
| 枪尖自绘贴图 `SpearTipTrail.png` 做大面积轨迹 | 容易出现白块或成片光幕。 |
| 直接采样武器贴图作为挥扫颜色 | 像素细节被拉伸，颜色杂、脏、花。 |
| `SlashTex` 同时作为形状和颜色 | 结果接近纯白，武器差异不明显。 |

## 9. 原版弹幕处理

刀剑当前策略是：

```text
拦截原版 Shoot
-> 自定义攻击按节奏发出刀光
-> 在指定帧由 VanillaMeleeProjectileEmitter 手动补发原版弹幕
```

长矛也应使用同一思想，但不能照搬刀剑 emitter 的通用 fallback。

很多长矛的 `item.shoot` 就是原版矛本体 projectile。如果通用补发：

```csharp
Projectile.NewProjectile(... player.HeldItem.shoot ...);
```

会产生：

```text
自定义长矛连招 + 原版刺击长矛 同时出现
```

因此长矛版 emitter 的规则必须是：

```text
默认不补发任何 HeldItem.shoot
只补发显式列入表中的特殊 projectile
不确定的武器先不补发
```

建议新增：

```text
Content/Projectiles/VanillaSpearProjectileEmitter.cs
```

第一版可以先留空，只建立接口和调用点；等逐把确认特殊武器后再填映射。

- 特色是在 AI、Shoot、OnHit 还是 Kill 中触发。
- 是否需要在终结刺击、右键投掷、命中时补发。
- 补发伤害是否会被当前倍率异常放大。

## 11. 命中和物品效果

`SpearStrikeProjectile.OnHitNPC` 当前调用：

```csharp
ItemLoader.OnHitNPC(sourceItem, player, target, in hit, damageDone);
```

这可以保留一部分物品命中逻辑。

但要注意：

- 如果某把武器的特色在原版 projectile AI 中产生，仅调用 `ItemLoader.OnHitNPC` 不一定够。
- 如果特色由原版 projectile 的 `OnHitNPC`、`AI` 或 `Kill` 触发，需要在长矛系统中显式重写或补发。
- 当前命中闪光和命中音效已经可复用，第一版不需要逐把变化。

## 12. 多人同步

当前 `SpearStrikeProjectile` 和 `SpearTrailGlowProjectile` 已同步：

```text
weaponItemType
comboStepIndex
branch
aimRotation
weaponLength
totalLifetimeUpdates
age
```

`SpearStrikeProjectile` 还同步 `_playedSwingSound`，避免挥舞音效重复触发。

视觉 profile 应由客户端根据 `weaponItemType` 本地解析，不要把贴图路径放进网络同步。

只要所有客户端拥有同一份白名单和视觉表，就能得到相同配色。

## 13. 实施步骤

### 步骤 1：扩展武器白名单

- 修改 `SpearWeaponClassifier`。
- 填入 19 把长矛白名单。
- 不做自动识别。

### 步骤 2：接入视觉配色

- 新增或等价实现 `SpearVisualProfile`。
- 新增或等价实现 `SpearVisualProfileResolver`。
- 绑定每把武器的配色源贴图和枪尖光颜色。
- `SpearTrailGlowProjectile` 通过 `_weaponItemType` 获取颜色数据。
- 找不到视觉 profile 时 fallback 到当前三叉戟配色。

### 步骤 3：保留现有动作基底

- 不复制 `SpearActionScheme`。
- 不为每把武器新增动作 step。
- 不把每把武器的宽度、长度、动作倍率塞进视觉 profile。

### 步骤 4：长矛版原版弹幕 emitter

- 新增 `VanillaSpearProjectileEmitter`。
- 默认不发任何 `HeldItem.shoot`。
- 只处理显式特殊武器。
- 第一版可以没有任何实际补发逻辑。

### 步骤 5：逐把验证

先验证 A 类普通武器，再验证 B 类特殊武器。

## 14. 验收清单

每把武器至少确认：

- 左键实际是三段循环：上挑、回扫、终结。
- 连招重置后从上挑开始。
- 手持武器贴图显示正确。
- 挥扫残影使用对应配色源贴图。
- 枪尖光颜色与残影主色一致。
- 不出现原版矛本体和自定义矛叠加。
- 命中 NPC 能造成伤害。
- 挥舞音效在 release 附近触发，而不是起手瞬间触发。
- 命中音效和命中闪光正常。
- 右键蓄力投掷不会丢失武器贴图。
- 多人客户端能看到相同方向和相同配色。

特殊武器额外确认：

- 原版特色是否保留。
- 原版特色是否重复。
- 特殊 projectile 是否发射过频。
- 特殊 projectile 伤害是否被异常放大。

## 15. 第一版安全底线

第一版必须满足：

```text
明确 19 把长矛白名单
普通左键按当前三段循环实现
复用 SpearActionScheme
禁用原版 Shoot
不通用补发 HeldItem.shoot
视觉配色按 weaponItemType 解析
挥扫继续使用 Mhd + SlashTex + 干净配色源贴图
不确定的特殊弹幕先不补发
```

宁可第一版少保留几个特殊弹幕，也不要出现原版矛本体和自定义连招叠加。
