# 全长矛适配技术文档

## 1. 目标

在当前三叉戟连招已经跑通的基础上，将长矛类武器批量接入同一套四段枪势动作。

本轮目标是扩大可用武器范围，不追求逐把武器的动作差异化。

统一采用当前连招：

```text
扎 -> 上挑 -> 绕身回扫 -> 地面进尺 / 空中抡枪下砸
```

## 2. 本轮明确取舍

本轮先接受以下统一处理：

- 武器长度和贴图原点差异先不单独处理。
- 宽度和动作强度先统一。
- 所有长矛先共用当前三叉戟连招。
- 原版弹幕按刀剑系统的思路手动补发。
- 特殊情况单独列入显式分支，不做通用猜测。

这意味着第一版实现的重点是稳定接入，不是逐把武器精修。

## 4. 现有长矛管线

当前三叉戟实现的主要模块：

| 模块 | 当前职责 |
| --- | --- |
| `SpearGlobalItem` | 判断武器是否接管，隐藏原版使用图，创建长矛 channel。 |
| `SpearChannelProjectile` | 读取鼠标方向，维持使用状态，推进长矛连段，发出 strike/trail。 |
| `SpearStrikeProjectile` | 负责实际伤害、碰撞、命中反馈、手持武器绘制。 |
| `SpearTrailGlowProjectile` | 负责枪杆残影、挥扫弧面、枪尖光。 |
| `TridentSpearComboScheme` | 当前唯一的长矛四段动作参数。 |
| `SpearMotion` | 根据动作、分支、进度计算枪柄和枪尖位置。 |
| `SpearCollisionEnvelope` | 统一碰撞宽度、枪尖延伸、视觉枪尖光尺度。 |

本轮批量适配不应复制一套动作系统。应该让所有武器先共用这些模块，再通过 profile 填配色和少量特例。

## 5. 建议新增数据结构

新增一个长矛 profile，不复用刀剑的 `WeaponSlashProfile`。

建议命名：

```text
Core/Spears/SpearVisualProfile.cs
Core/Spears/SpearProfileResolver.cs
```

`SpearVisualProfile` 第一版只需要包含视觉和原版弹幕补发相关数据：

```csharp
public readonly struct SpearVisualProfile
{
    public readonly int ItemType;
    public readonly string SweepColorTexturePath;
    public readonly Color TipGlowColor;
    public readonly SpearVanillaProjectileMode VanillaProjectileMode;
}
```

暂时不要把长度、宽度、动作倍率放进去。当前决策是这些先统一。

`SpearProfileResolver` 负责：

- 判断某个 `ItemID` 是否支持长矛接管。
- 返回配色源贴图路径。
- 返回枪尖光颜色。
- 返回是否需要补发原版特色弹幕。

## 6. 入口接管规则

当前 `SpearGlobalItem.ShouldUseSpearAction` 只允许 `ItemID.Trident`。

批量适配时应改成显式白名单：

```text
SpearProfileResolver.TryGetProfile(item.type, out _)
```

不要使用以下方式自动判断：

```text
item.useStyle == ItemUseStyleID.Rapier
item.shoot > ProjectileID.None
item.noMelee
item.channel
```

原因是这些条件会误伤其他非标准长柄/投射武器。

接管后仍保持：

```text
item.noUseGraphic = true
item.noMelee = true
item.channel = true
item.useStyle = ItemUseStyleID.Rapier
```

但 `item.shoot` 的处理要和弹幕补发设计配合，见下一节。

## 7. 原版弹幕处理

### 7.1 刀剑当前策略

刀剑不是保留原版 `Shoot`，而是：

```text
拦截原版 Shoot
-> 自定义攻击节奏发出刀光
-> 在合适帧由 VanillaMeleeProjectileEmitter 手动补发原版弹幕
```

长矛也应使用同一思想，但不能照搬刀剑的通用 fallback。

### 7.2 长矛的关键差异

很多长矛的 `item.shoot` 就是原版矛本体 projectile。

如果通用补发：

```csharp
Projectile.NewProjectile(... player.HeldItem.shoot ...)
```

会产生严重问题：

```text
自定义长矛连招 + 原版刺击长矛 同时出现
```

这不是保留特色，而是双武器叠加。

因此长矛必须禁止通用 fallback。

### 7.3 长矛版 emitter 原则

建议新增：

```text
Content/Projectiles/VanillaSpearProjectileEmitter.cs
```

第一版规则：

```text
默认不补发任何 HeldItem.shoot
只有显式列入 profile 的武器才补发特殊弹幕
不确定的武器先只保留自定义连招和视觉
```

也就是说，长矛版 emitter 不能有刀剑里的这类兜底逻辑：

```csharp
if (player.HeldItem.shoot > ProjectileID.None)
{
    Projectile.NewProjectile(... player.HeldItem.shoot ...);
}
```

### 7.4 `item.shoot` 清理注意事项

当前 `SpearGlobalItem.SetDefaults` 直接执行：

```csharp
item.shoot = ProjectileID.None;
```

如果后续需要读取某把武器的原版 `shoot`，这里会导致运行时从 `player.HeldItem.shoot` 读不到原值。

因此补发特殊弹幕时不要依赖 `HeldItem.shoot`。

应采用显式映射：

```text
ItemID.NorthPole -> ProjectileID.NorthPoleSpear / 相关特效 projectile
ItemID.GhastlyGlaive -> 对应原版特殊 projectile
...
```

具体 projectile ID 需要逐把查证后再填，不能猜。

## 8. 弹幕补发时机

第一版不要每段都补发特殊弹幕。

建议规则：

| 动作段 | 补发策略 |
| --- | --- |
| 第 1 段 扎 | 适合短直线特色弹幕。 |
| 第 2 段 上挑 | 默认不补发。 |
| 第 3 段 回扫 | 默认不补发，避免范围失控。 |
| 第 4 段 地面进尺 | 适合主力直线弹幕或强化弹幕。 |
| 第 4 段 空中下砸 | 可在落点或枪尖终点补发特殊效果，但第一版可先不做。 |

建议第一版只在第 1 段和第 4 段考虑补发。

## 9. 特殊武器分流

第一版可分三类：

### A. 普通接管，无原版弹幕补发

适用于大多数普通矛和矿物矛。

```text
Spear
Trident
Swordfish
The Rotted Fork
Dark Lance
Cobalt Naginata
Palladium Pike
Mythril Halberd
Orichalcum Halberd
Adamantite Glaive
Titanium Trident
Gungnir
Chlorophyte Partisan
Obsidian Swordfish
Slime Spear
```

这些武器第一版只需要连招、碰撞、手持绘制和配色 profile。

### B. 接管，但特殊弹幕待确认

```text
Storm Spear
Mushroom Spear
Ghastly Glaive
North Pole
```

这些武器第一版可以先接管动作，但不要补发未知 projectile。

后续逐把确认：

- 原版特色是否来自 `item.shoot`。
- 原版特色是否来自 projectile AI。
- 是否需要在第 1 段、第 4 段或命中时补发。
- 是否需要保留原版 debuff 或 OnHit 行为。

### C. 暂不纳入

骑枪类和非 `Category:Spears` 武器暂不纳入。

## 10. 视觉 profile 接入

当前 `SpearTrailGlowProjectile` 使用：

```text
Textures[0] = SlashTex
Textures[1] = SpearSweepColor
```

批量适配后应改为：

```text
Textures[0] = SlashTex
Textures[1] = profile.SweepColorTexture
```

枪尖光当前是单一常量：

```csharp
SpearTipGlowColor
```

批量适配后应改为：

```text
profile.TipGlowColor
```

建议所有 profile 的颜色都来自同一张设计表，避免残影和枪尖光不统一。

第一版仍然不要恢复武器贴图采样作为残影颜色源。之前已经验证过：

```text
武器贴图采样 -> 颜色杂、脏、像素细节被拉伸
SlashTex 采样 -> 纯白
干净配色源贴图 -> 更适合
```

## 11. 命中和 OnHit

`SpearStrikeProjectile.OnHitNPC` 当前已经调用：

```csharp
ItemLoader.OnHitNPC(sourceItem, player, target, in hit, damageDone);
```

这能保留一部分物品命中逻辑。

但注意：

- 如果某武器特色依赖原版 projectile AI 的命中逻辑，仅调用 `ItemLoader.OnHitNPC` 不一定足够。
- 如果特色是 projectile 自己在 AI 或 OnHit 里生成的，必须在长矛系统里手动补发或重写。
- 命中粒子和命中闪光第一版可以继续复用当前逻辑。

## 12. 多人同步

视觉 profile 必须由 `weaponItemType` 在客户端本地解析，不需要把贴图路径通过网络同步。

当前 `SpearStrikeProjectile` 和 `SpearTrailGlowProjectile` 已同步：

```text
weaponItemType
comboStepIndex
branch
aimRotation
weaponLength
totalLifetimeUpdates
age
```

只要所有客户端拥有同一份 profile 表，就可以通过 `weaponItemType` 得到相同配色。

如果后续引入随机弹幕、随机粒子或特殊落点，才需要额外同步随机种子或关键参数。

## 13. 实施步骤

推荐按以下顺序实现。

### 步骤 1：新增 profile 数据

- 新增 `SpearVisualProfile`。
- 新增 `SpearProfileResolver`。
- 填入 19 把武器白名单。
- 绑定每把武器的配色源贴图和枪尖光颜色。

### 步骤 2：入口改为 profile 白名单

- `SpearGlobalItem.ShouldUseSpearAction` 改为查 `SpearProfileResolver`。
- 保持当前接管行为。
- 不做自动识别。

### 步骤 3：视觉读取 profile

- `SpearTrailGlowProjectile` 通过 `_weaponItemType` 获取 profile。
- 挥扫残影绑定对应 `SweepColorTexture`。
- 枪尖光使用对应 `TipGlowColor`。
- 找不到 profile 时 fallback 到三叉戟配色。

### 步骤 4：长矛版原版弹幕 emitter

- 新增 `VanillaSpearProjectileEmitter`。
- 默认不发任何 `HeldItem.shoot`。
- 只处理显式特殊武器。
- 先留空或只做已确认特例。

### 步骤 5：在 channel 或 strike 中调用 emitter

建议在真正发出 `SpearStrikeProjectile` 的同一帧调用。

第一版只允许：

```text
第 1 段
第 4 段
```

不要在第 2/3 段默认补发弹幕。

### 步骤 6：逐把验证

先验证 A 类普通武器，再验证 B 类特殊武器。

## 14. 验收清单

每把武器至少确认：

- 左键能进入四段连招。
- 第 4 段地面/空中分支仍正常。
- 手持武器贴图显示正确。
- 残影颜色使用对应配色源贴图。
- 枪尖光颜色与残影统一。
- 不出现原版矛本体和自定义矛叠加。
- 命中 NPC 能造成伤害。
- 命中音效和命中闪光正常。
- 多人客户端能看到相同方向和相同配色。

特殊武器额外确认：

- 原版特色是否保留。
- 原版特色是否重复。
- 弹幕发射频率是否过高。
- 特殊弹幕伤害是否被异常放大。

## 15. 第一版安全底线

第一版必须满足以下底线：

```text
明确 19 把长矛白名单
禁用原版 Shoot
不通用补发 HeldItem.shoot
所有武器使用当前连招
视觉 profile 按 weaponItemType 解析
不确定的特殊弹幕先不补发
```

宁可第一版少保留几个特殊弹幕，也不要出现原版矛本体和自定义连招叠加。
